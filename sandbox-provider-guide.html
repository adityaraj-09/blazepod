Architecture overview
A sandbox provider is a multi-layer system. Each layer handles a specific isolation primitive. Understanding the full stack before writing code is critical — wrong layer abstractions compound into performance debt that's very hard to unwind.

02
MicroVM layer — Firecracker
Firecracker is the foundation. It's a KVM-based VMM (Virtual Machine Monitor) written in Rust by AWS for Lambda and Fargate. It boots a stripped Linux kernel in ~125ms cold, exposing only the minimum device model: virtio-net, virtio-block, virtio-vsock, and a serial console. No USB, no PCI hotplug, no BIOS emulation — that's the security surface reduction.
How KVM + Firecracker works
KVM (Kernel-based Virtual Machine) exposes /dev/kvm — a Linux kernel module that lets userspace programs create and manage VMs using CPU hardware virtualization extensions (Intel VT-x / AMD-V). Firecracker is a KVM client: it opens /dev/kvm, allocates guest physical memory as a mmap'd region, loads a Linux kernel image, sets up virtio devices, and runs the VM. Each Firecracker process = one microVM. The jailer binary further restricts the Firecracker process via seccomp and chroot.
// Simplified: what Firecracker does internally (concept)
let kvm = Kvm::new()?;                          // open /dev/kvm
let vm  = kvm.create_vm()?;                     // ioctl KVM_CREATE_VM
let mem = GuestMemoryMmap::new(regions)?;       // mmap guest RAM
vm.set_user_memory_region(mem)?;                // map into VM
let vcpu = vm.create_vcpu(0)?;                  // virtual CPU
// load kernel into guest memory, set registers, run
vcpu.run()?;                                     // KVM_RUN ioctl — blocks until VM exit
Snapshot / restore for cold start reduction
Firecracker supports pausing a VM and dumping its full state (memory + device state) to disk. On restore, it mmaps the snapshot file and resumes from exactly that point. This is how you get sub-30ms starts: snapshot a VM that's already booted with your base image loaded, store it on NVMe, restore it per request. The writable layer is new per sandbox (overlayfs), so the snapshot is shared read-only.
# Firecracker API — create snapshot
PUT /snapshot/create
{
  "snapshot_type": "Full",
  "snapshot_path": "/snapshots/python312-base.snap",
  "mem_file_path":  "/snapshots/python312-base.mem"
}

# Restore from snapshot (new VM, same state)
PUT /snapshot/load
{
  "snapshot_path": "/snapshots/python312-base.snap",
  "mem_backend": { "backend_type": "File", "backend_path": "/snapshots/python312-base.mem" },
  "enable_diff_snapshots": true
}

03
Container runtime + Linux namespaces
Inside the microVM, you run a minimal init process and container runtime. Understanding what the kernel actually does for isolation is essential — you can't debug namespace issues without this mental model.
The 7 namespace types
Linux namespaces virtualize system resources. Each sandbox gets its own instance of each namespace, so processes inside see an isolated view of the system.
PID namespace
Process tree starts at PID 1. Sandbox can't see or signal host processes.
Net namespace
Own network stack: interfaces, routing table, iptables rules, socket table.
Mount namespace
Independent filesystem view. overlayfs provides per-sandbox writable layer.
User namespace
UID/GID remapping. Root inside = unprivileged user outside.
IPC namespace
Isolates System V IPC and POSIX message queues.
UTS namespace
Own hostname and domain name. Prevents hostname leaks.
Time namespace
Independent CLOCK_BOOTTIME offset. (Linux 5.6+)
cgroup v2 — resource enforcement
cgroups limit and account for resource usage. v2 (unified hierarchy) is the right choice — it has better semantics for nested containers and delegate mode lets rootless runtimes manage sub-hierarchies.
# Create a cgroup for a sandbox
mkdir /sys/fs/cgroup/sandboxes/sb-abc123

# Set limits
echo "500000 1000000" > /sys/fs/cgroup/sandboxes/sb-abc123/cpu.max   # 50% of 1 CPU
echo "536870912"     > /sys/fs/cgroup/sandboxes/sb-abc123/memory.max  # 512 MiB
echo "134217728"     > /sys/fs/cgroup/sandboxes/sb-abc123/memory.swap.max

# OOM event notification (watch memory.events via inotify in your host agent)
cat /sys/fs/cgroup/sandboxes/sb-abc123/memory.events
# → oom 3   (means this cgroup got OOM-killed 3 times)
seccomp — syscall allowlisting
Even with namespace + cgroup isolation, a sandbox can make arbitrary syscalls. seccomp-bpf filters which syscalls are allowed. Your profile should allowlist ~60–80 syscalls (the minimum for Python/Node to run) and block everything else with SIGKILL (not SIGSYS — attackers can catch SIGSYS).
// seccomp profile skeleton (libseccomp in Go via goseccomp)
filter := seccomp.NewFilter(seccomp.ActKill)   // default: kill
// allow only what's needed
for _, sc := range []string{
    "read", "write", "openat", "close", "fstat", "mmap",
    "mprotect", "munmap", "brk", "exit_group", "futex",
    "clone", "wait4", "execve", /* ... ~60 more */
} {
    filter.AddRule(seccomp.ScmpSyscall(sc), seccomp.ActAllow)
}
filter.Load()  // installs the BPF program into the kernel

04
Orchestrator internals
The orchestrator is the central nervous system. It's a stateful Go service that owns the scheduler, VM pool manager, and the sandbox state machine. It communicates with host agents over gRPC.
Admission → scheduling flow
1
Request arrives at API gateway
JWT validated, tenant resolved, sandbox spec parsed and validated against JSON schema. Quota checked atomically in Redis (INCR with EXPIRE). If over quota → 429.
2
Enqueue to priority queue
Internal min-heap keyed by (deadline, priority). Deadline = now + client timeout. Priority = plan tier weight × recency factor.
3
Scheduler dequeues + scores hosts
For each candidate host: score = 0.4×cpu_fit + 0.35×warm_pool_hit + 0.25×host_health. Highest scorer wins. O(n_hosts) per decision — keep host list <200 per region.
4
Host agent receives placement
gRPC call to host agent: PlaceSandbox(spec). Agent picks an idle VM from local pool, applies the writable layer, configures the network namespace, returns sandbox handle.
5
Sandbox ID returned to client
Client gets sandbox_id + exec endpoint (vsock or HTTP on localhost within the VM). State machine: queued → provisioning → running.
VM pool manager — demand prediction
// Pool target: EWMA-based demand prediction
type PoolManager struct {
    ewmaRPS      float64
    p95DurSec    float64
    currentPool  int
}

func (p *PoolManager) TargetSize() int {
    // Little's Law: avg concurrency = arrival_rate × avg_duration
    concurrency := p.ewmaRPS * p.p95DurSec
    headroom     := concurrency * 1.3
    floor        := 10
    return max(int(headroom), floor)
}

func (p *PoolManager) UpdateEWMA(newRPS float64) {
    alpha    := 0.3   // smoothing factor — tune per workload
    p.ewmaRPS = alpha*newRPS + (1-alpha)*p.ewmaRPS
}
State machine — enforcement rules
State transitions must be atomic and append-only in the audit log. Use a compare-and-swap on the state field (etcd or Postgres with optimistic locking). Illegal transitions (e.g. running → queued) must be rejected at the store level, not just application level.
var validTransitions = map[State][]State{
    Queued:       {Provisioning, Failed},
    Provisioning: {Running, Failed},
    Running:      {Draining, Failed},
    Draining:     {Terminated},
    Terminated:   {},  // terminal — no transitions out
    Failed:       {},  // terminal
}

func Transition(from, to State) error {
    for _, allowed := range validTransitions[from] {
        if allowed == to { return nil }
    }
    return fmt.Errorf("illegal transition %s→%s", from, to)
}

05
Networking — veth, eBPF, WireGuard
Networking is where most sandbox providers leak tenant isolation. The goal: each sandbox gets its own network stack, egress is filtered at line rate, and there's no path for traffic to traverse between sandboxes.
veth pair + network namespaces
A veth (virtual ethernet) pair is a linked pair of virtual interfaces — packets in one end come out the other. You put one end inside the sandbox net namespace and keep one on the host. Traffic flows through the pair, and the host end connects to a bridge or is routed directly.
# Setup per-sandbox veth pair (host agent does this in Go via netlink)
ip link add veth-sb0 type veth peer name veth-sb0-host
ip link set veth-sb0 netns /proc/<sandbox-pid>/ns/net   # move into sandbox netns
ip netns exec sb0 ip addr add 10.0.1.2/24 dev veth-sb0   # sandbox IP
ip addr add 10.0.1.1/24 dev veth-sb0-host                # host side
ip link set veth-sb0-host up
ip netns exec sb0 ip route add default via 10.0.1.1      # default route via host
eBPF for egress filtering
eBPF programs attached at TC (Traffic Control) layer inspect packets at line rate without entering userspace. For per-sandbox egress policy (allowlist of destination IPs/hostnames), you write a BPF program that parses the IP header, checks against a BPF map (hash map of allowed destinations), and either passes or drops the packet.
// eBPF TC egress program (C, compiled with clang -target bpf)
struct bpf_map_def SEC("maps") allowed_dsts = {
    .type        = BPF_MAP_TYPE_LPM_TRIE,     // longest prefix match — for CIDR allowlists
    .key_size    = sizeof(struct bpf_lpm_trie_key) + 4,
    .value_size  = sizeof(__u32),
    .max_entries = 1024,
};

SEC("tc/egress")
int sandbox_egress(struct __sk_buff *skb) {
    void *data     = (void *)(long)skb->data;
    void *data_end = (void *)(long)skb->data_end;
    struct iphdr *ip = data + sizeof(struct ethhdr);
    if ((void *)(ip + 1) > data_end) return TC_ACT_SHOT;

    // LPM lookup on destination IP
    struct { __u32 prefixlen; __u32 addr; } key = { 32, ip->daddr };
    if (!bpf_map_lookup_elem(&allowed_dsts, &key)) return TC_ACT_SHOT;
    return TC_ACT_OK;
}
WireGuard for tenant-level isolation
For enterprise multi-tenancy, each tenant gets a WireGuard tunnel. All sandbox traffic for a tenant is routed through their tunnel endpoint. This adds cryptographic isolation on top of the eBPF filtering — even if eBPF is misconfigured, traffic between tenants can't be read without the WireGuard keys.

Storage — overlayfs + snapshots
overlayfs — layered filesystem
overlayfs stacks filesystem layers. A shared, read-only lower layer (your base image) is combined with a per-sandbox writable upper layer. Reads hit the upper first, then fall through to the lower. Writes go to the upper only. The lower layer is shared across all sandboxes using the same image — significant RAM/disk savings.
# Mount an overlay filesystem for sandbox sb-abc123
mount -t overlay overlay \
  -o lowerdir=/images/python312-base,\
     upperdir=/sandboxes/sb-abc123/upper,\
     workdir=/sandboxes/sb-abc123/work \
  /sandboxes/sb-abc123/rootfs

# After sandbox exit — wipe the upper layer only
rm -rf /sandboxes/sb-abc123/upper/*
# lower dir untouched — reused by next sandbox
QCOW2 snapshots for VM-level state
QCOW2 (QEMU Copy-On-Write v2) is the disk format used for Firecracker block device snapshots. A base QCOW2 image contains the kernel + base rootfs. Per-sandbox QCOW2 files use the base as a backing file — only diffs are stored. This is the disk equivalent of overlayfs.
# Create base image
qemu-img create -f qcow2 base-python312.qcow2 2G

# Create per-sandbox disk backed by base (copy-on-write)
qemu-img create -f qcow2 -b base-python312.qcow2 -F qcow2 \
  sb-abc123.qcow2
Persistent sandbox state — object store
For agent workloads that need to persist state between runs (files written, packages installed), serialize the writable overlay layer to S3/R2 as a tar.zst archive after execution. On next run, restore it before mounting. This is what e2b's filesystem persistence does. Use zstd for compression — it's significantly faster than gzip at similar ratios.

Control plane API
External API — REST + WebSocket
The public-facing API is a REST API for sandbox lifecycle (create, list, kill) and WebSocket for streaming exec output. Use HTTP/2 for multiplexing. Authentication via API keys (HMAC-SHA256 signed) or JWTs. Rate limiting per tenant via token bucket in Redis.
# SDK usage (what your users write)
const sb = await client.sandboxes.create({
  image:   "python:3.12",
  cpu:     0.5,
  memory:  512,
  timeout: 30_000,
  egress:  ["api.openai.com", "pypi.org"],
})

const result = await sb.exec("python3 -c 'print(42)'")
console.log(result.stdout)  // "42\n"

await sb.kill()
Internal API — gRPC between orchestrator and host agents
All internal communication uses gRPC with protobuf. This gives you typed contracts, bidirectional streaming for log drains, and easy TLS mutual auth between services. Define your proto schema first — it becomes the contract between every internal service.
// orchestrator.proto (excerpt)
service HostAgent {
  rpc PlaceSandbox   (PlaceRequest)   returns (PlaceResponse);
  rpc TerminateSandbox(TerminateReq)  returns (TerminateResp);
  rpc StreamLogs      (LogRequest)    returns (stream LogChunk);
  rpc Heartbeat       (HeartbeatReq) returns (HeartbeatResp);
}

message PlaceRequest {
  string sandbox_id    = 1;
  string image_ref     = 2;
  uint32 cpu_milli     = 3;
  uint32 mem_mib       = 4;
  uint32 timeout_ms    = 5;
  repeated string egress_allowlist = 6;
  string snapshot_key  = 7;   // which pool snapshot to restore from
}
Exec inside sandbox — vsock
vsock (VM socket) is a high-performance socket type for VM↔host communication that doesn't go through the network stack. Inside the VM, your agent listens on a vsock port. The host connects via AF_VSOCK to send exec requests and stream stdio. This is faster than routing through the veth pair and stays isolated from tenant network traffic.

Structured logging + audit trail
Every state transition emits a structured log event with: sandbox_id, tenant_id, old_state, new_state, host_id, timestamp_ns, reason. These are the source of truth for billing and debugging. Write to a durable append-only store (Kafka → ClickHouse or S3 for long-term).
Distributed tracing — OpenTelemetry
Instrument every gRPC call and critical code path with OpenTelemetry spans. A single sandbox request spans: API handler → quota check → scheduler decision → host agent PlaceSandbox → VM restore → exec. Without tracing, latency regressions are nearly impossible to root-cause across services.

Security model
Defense in depth — layers must be independent
No single layer should be the only thing between tenant code and the host. The security model is: microVM hardware isolation → namespace isolation → seccomp syscall filtering → cgroup resource limits → eBPF network policy. A vulnerability in one layer should not give an attacker the next layer.
Critical: writable layer wipe
Before any sandbox slot is reused, the upper overlayfs layer must be fully wiped and the new overlayfs remounted. A missed wipe leaks previous tenant's written files. Use a cryptographic check (hash the empty upper dir) before marking a VM as idle.
STONITH before slot reassignment
Never reassign a sandbox slot from a host you can't reach. Use out-of-band IPMI/BMC to force-power-cycle the host before considering its slots free. This prevents split-brain state where two tenants think they own the same VM.
Image supply chain
Only allow OCI images from your own registry. Verify image digests (sha256) before mounting as the lower layer. Use ORAS (OCI Registry as Storage) to store and distribute base images with content-addressable integrity guarantees. Never pull from Docker Hub directly at runtime — cache in your registry.

Language choices — and why
Go — orchestrator + host agent + API
Best concurrency model for managing thousands of long-lived goroutines (one per sandbox). Low GC pauses (<1ms with tuning). Excellent netlink, gRPC, and kernel interface libraries. Fast compile times for iteration.
Rust — hot path components
Use for the VM agent inside each microVM (the process that handles exec/stdio). Zero-cost abstractions, no GC, predictable latency. Firecracker itself is Rust — contributes upstream skills. Also ideal for eBPF helpers and any custom VMM extensions.
C — eBPF programs
eBPF programs are written in restricted C and compiled with clang -target bpf. You can't avoid C here. Use libbpf for the loader. CO-RE (Compile Once, Run Everywhere) removes kernel version dependency.
TypeScript — public SDK + CLI
SDK that developers actually use. TypeScript gives you type safety in the generated client. Use Zod for runtime validation of API responses. Publish to npm. Also build a Go SDK for backend-heavy users.
Don't use Python for the data plane
Python's GIL and GC make it unsuitable for the orchestrator or host agent. It's fine for tooling, tests, and internal scripts, but any code in the request path should be Go or Rust.

Build roadmap — 3 phases
Phase 1 — Foundation (weeks 1–6)
Single-host proof of concept
  • Run Firecracker manually, understand VMM API
  • Write Go host agent: create/kill VMs via FC API
  • Implement overlayfs mount + teardown lifecycle
  • Basic cgroup v2 CPU + mem limits
  • vsock-based exec handler inside VM
  • Minimal REST API: create, exec, kill
  • Target: 200ms p99 cold start on one host
Phase 2 — Hardening (weeks 7–14)
Multi-host + pool + security
  • Snapshot / restore pipeline (<30ms warm starts)
  • VM pool manager with EWMA demand prediction
  • Multi-host orchestrator with gRPC host agents
  • eBPF egress filtering per sandbox
  • seccomp profile for common runtimes
  • Prometheus metrics + basic Grafana dashboards
  • Tenant quota enforcement (Redis)
  • Target: 30ms p99 warm start, 3-nines uptime
Phase 3 — Scale (weeks 15–24)
Production-grade operations
  • STONITH fault tolerance + host health scoring
  • OpenTelemetry distributed tracing
  • OCI image registry + ORAS distribution
  • Filesystem persistence (tar.zst → S3)
  • WireGuard per-tenant network isolation
  • TypeScript + Go SDK releases
  • Load testing: 10k concurrent sandboxes
  • Target: <15ms p99 warm start at scale
